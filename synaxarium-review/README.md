# Kalendar Review — An Anglican Synaxarium

A day-by-day review tool for the Kalendar v0.1 candidate matrices: see every ranked candidate for a given civil date, its SIN, its full sourcing and rationale, and any cross-date harmonization warnings — then concur, override, or record something else entirely, with a comment, and move on.

This is the third build of this tool. It is regenerated fresh from the current `origin/main` state of `jwjeffery-prayerappnew`, after the repo-side repairs in commit `16747d5` (and the earlier `864effd`, `57c03e9`). It does not carry forward any stale data from earlier sessions — see the validation report for exactly what was checked.

## Files

```
synaxarium-review/
  index.html                        the page
  styles.css                        the look
  app.js                            the behavior
  data/
    kalendar-data.json               generated by build_data_v3.py from the current repo
  build_data_v3.py                   the build/validation script (rerun this whenever the repo changes)
  merge_decisions_into_csv.py        offline script to apply confirmed decisions back into the monthly CSVs
  validation-report.md               the validation run this delivery is based on
```

## Setup

Double-clicking `index.html` mostly works but some browsers block loading `data/kalendar-data.json` from a `file://` path. Best options:

**GitHub Pages:** add this folder to the repo, then in **Settings → Pages**, deploy from the branch/folder it lives in. You'll get a URL like `https://jwjeffery.github.io/PrayerAppNew/synaxarium-review/`.

**universaloffice.com:** drop the folder in wherever the site serves static files; it runs at whatever path you place it.

**Quick local test:** from a terminal, `cd` into the folder and run `python3 -m http.server 8000`, then open `http://localhost:8000`.

## Using it

- Left rail: all twelve months (all built now — nothing grayed out), each day marked with a status dot (hollow = pending, filled green = decided).
- Main panel: one day, every candidate for it in rank order. Each card shows candidate name, **SIN** (prominently, in its own row — a plain black badge for a confirmed SIN, a red "no SIN found" badge if the join ever fails again), rank, designation, year/period, a tradition tag color-coded **blue for Anglican-sourced material and gold for Great-Church-sourced material**, source witnesses, source tier, preliminary status, ranking rationale, review flags, and notes. Any cross-date harmonization flag for that specific candidate appears directly on its card, in a red-bordered block; if a harmonization entry can't be tied to one candidate after clean parsing, it appears instead as a banner at the top of the day (the current data has zero of these — every entry resolved to a specific card).
- Click a card to select it, use "Concur with rank 1" for the common case, or toggle "None of the above" to record something not listed, with a comment box always available.
- "Confirm & next day" records the decision and advances automatically. Revisiting a day (clicking back to it in the left rail) shows your existing decision and lets you change it — nothing is locked in.

## Disk persistence

**In Chrome or Edge:** click **"Connect decisions file"** once. A native save dialog lets you create a new `synaxarium-decisions.json` or pick an existing one to keep adding to. From then on, every confirmed day writes straight to that file on disk — not just the browser cache — and the tool reconnects automatically next time (occasionally asking you to re-approve access after a browser restart, a normal security check; the button will say "Reconnect decisions file" when that's needed).

**In Safari or Firefox:** direct disk-writing isn't available in those browsers at all. The button is disabled with an explanation. Decisions still autosave to the browser's local storage, and you should use **Export decisions** regularly as your real save file.

**Import decisions** merges an exported file back in — by comparing timestamps per day, not blind overwrite — so re-connecting an older file won't stomp on newer decisions already recorded.

## Exported decision format

Each day's export includes: month, date, selected candidate, selected SIN (if applicable), decision type (`concur_rank1` / `select_alternate` / `custom`), reviewer comments, timestamp, and a full snapshot of the original candidate row (including its SIN and any harmonization warnings) — everything `merge_decisions_into_csv.py` needs to apply the decision back to the source CSV later.

## Merging decisions back into the CSVs

The static UI never writes to the source CSVs — only to your own decisions file. When you're ready to apply a batch of confirmed decisions to the monthly candidate files:

```
python3 merge_decisions_into_csv.py \
    --decisions synaxarium-decisions-2026-07-04.json \
    --kalendar-dir /path/to/PrayerAppNew/data/kalendar \
    --out-dir ./merged_output
```

This writes **updated copies** into `--out-dir` (never touching your real repo unless you explicitly pass `--in-place`), setting `decision_status=Decided` and `final_primary=<your choice>` on the matching row for each `concur_rank1` / `select_alternate` decision. "None of the above" decisions aren't mechanically merged — they're written to `custom-decisions-needing-manual-entry.md` in the output folder, since adding a new row is a content decision, not a mechanical edit, and this script doesn't fabricate liturgical text or rows on its own. Review the diff between `--out-dir` and your real files before committing anything.

## Regenerating the data file

Whenever the repo's candidate CSVs, SIN maps, or harmonization ledger change, rerun the build script against a fresh checkout:

```
python3 build_data_v3.py --kalendar-dir /path/to/PrayerAppNew/data/kalendar --out-dir ./synaxarium-review-out
```

This regenerates `data/kalendar-data.json` and `validation-report.md` together. The script is strict by design: it does not silently repair malformed rows, missing SINs, or excluded figures reappearing — it reports them as hard failures instead, so bad data is never quietly packaged into a working-looking UI.
